# ORBIT PROTOCOL 프로젝트 인수인계 문서

## 0. 문서 목적

이 문서는 웹 기반 1인용 인지·집중 훈련 게임 서비스 **ORBIT PROTOCOL**의 기획, 게임 규칙, 사용자 계정, 진척도, 랭킹, 저장 구조, Cloudflare 배포 구조를 한 번에 인수인계하기 위한 문서다.

이 프로젝트는 놀이치료, 인지 훈련, 실행기능 훈련, 작업기억, 인지 전환 등의 개념에서 영감을 받았지만, 의료행위나 실제 치료를 제공하는 서비스가 아니다.

서비스 내에는 다음과 같은 취지의 안내 문구를 명시한다.

> ORBIT PROTOCOL은 주의집중, 작업기억, 계획, 인지 전환 능력을 게임 형태로 연습하는 일반 웰니스·교육 콘텐츠입니다. 의료행위, 진단 또는 치료를 제공하지 않습니다.

# 1. 프로젝트 개요

## 1.1 프로젝트명

**ORBIT PROTOCOL**

## 1.2 한 줄 소개

고장 난 장거리 탐사선의 세 관제 구역을 복구하며 계획, 작업기억, 인지 전환 능력을 사용하는 1인용 라운드 게임 서비스.

## 1.3 핵심 콘셉트

플레이어는 장거리 탐사선의 신입 관제 요원이다.

탐사선은 우주 폭풍을 통과하는 과정에서 자동 시스템과 중앙 관제 장치 일부가 손상되었다. 플레이어는 우주선 내부의 세 관제 구역을 직접 담당하며 각 구역에서 발생하는 문제를 해결한다.

세 게임은 각각 독립적으로 실행되며, 사용자가 상단 내비게이션에서 자유롭게 선택할 수 있다. 게임을 순서대로 강제 연결하지 않는다.

## 1.4 서비스 목표

- 10~20분 안에 끝나는 독립적인 라운드 게임 제공
- LLM, 생성형 AI, 실시간 멀티플레이 없이 완결되는 구조
- 사용자의 진행 상태와 성공률을 누적
- 닉네임 기반 랭킹으로 재방문 동기 제공
- 모바일과 데스크톱 모두에서 플레이 가능
- 무료 Cloudflare 인프라 안에서 최대한 오래 운영

# 4. 공통 세계관

플레이어가 탑승한 우주선은 장거리 탐사선이다.

우주선이 미확인 우주 폭풍을 통과하는 과정에서 중앙 자동화 시스템이 손상되었다. 자동 관제 시스템이 완전히 멈춘 것은 아니지만, 일부 판단과 분류를 사람이 직접 수행해야 하는 상태다.

플레이어는 다음 세 구역을 담당한다.

```text
COMMAND DECK
함교 작전실
제한된 자원으로 긴급 임무를 계획하고 처리한다.

CARGO CORE
화물 분류실
복잡한 규칙에 따라 화물과 표본을 분류한다.

VECTOR SHIFT
항로 전환실
계속 바뀌는 항법 기준에 맞춰 항로를 전환한다.
```

세 게임의 결과는 전체 탐사선 상태에 시각적으로 반영할 수 있다.

예시:

```text
함교 안정도
화물 안전도
항로 정확도
전체 항해 진행률
```

단, 이 상태값이 의료적 평가처럼 보이지 않게 한다.

# 5. 공통 게임 구조

## 5.1 공통 플레이 흐름

```text
게임 선택
→ 게임 소개
→ 현재 진행 상태 확인
→ 시작 라운드 선택
→ 라운드 시작
→ 규칙 또는 임무 확인
→ 사용자 입력
→ 결과 판정
→ 점수 계산
→ 라운드 결과
→ 자동 저장
→ 다음 라운드 또는 종료
```

## 5.2 공통 헤더 정보

플레이 중 화면 상단에 다음을 표시한다.

```text
현재 라운드 4 / 10
현재 점수 1,240
성공률 83%
남은 시간 00:38
일시정지
사운드
```

## 5.3 플레이 모드

초기 버전은 별도의 쉬움·보통·어려움 선택을 두지 않고, 라운드가 올라갈수록 자연스럽게 난도가 증가하게 구성한다.

필요할 경우 이후 다음 모드를 추가할 수 있다.

```text
짧은 항해
기본 항해
전체 항해
```

초기 MVP에서는 전체 라운드 구조만 구현해도 된다.

## 5.4 실패 처리

한 번의 실패로 게임 전체를 종료하지 않는다.

권장 흐름:

```text
이번 라운드 통과 실패
→ 현재 결과 표시
→ 같은 라운드 재도전
→ 이전 라운드 선택
→ 다른 게임 이동
```

피해야 할 문구:

```text
집중력이 부족합니다.
기억력이 낮습니다.
훈련이 필요합니다.
```

권장 문구:

```text
예외 규칙에서 실수가 많았습니다.
규칙 전환 직후 반응이 늦어졌습니다.
이번 라운드에는 방해 신호가 많았습니다.
```

# 6. GAME 1: COMMAND DECK

## 6.1 한글명

함교 작전실

## 6.2 게임 목적

제한된 시간, 전력, 승무원, 수리 부품 안에서 여러 우주선 임무를 처리하는 전략·계획 게임.

## 6.3 주요 훈련 요소

- 계획
- 우선순위 판단
- 실행기능
- 충동 억제
- 자원 관리
- 상황 변화에 대한 유연한 대응

## 6.4 최대 라운드

**8라운드**

예상 플레이 시간:

```text
전체 진행 15~20분
라운드당 약 1.5~3분
```

## 6.5 주요 자원

- 전력
- 승무원
- 시간
- 수리 부품
- 선체 안정도

## 6.6 임무 예시

### 산소 정화 장치 복구

```text
전력 2
승무원 1
소요 2턴
중요도 높음
실패 시 생명 유지도 감소
```

### 외부 통신 복구

```text
전력 2
수리 부품 1
소요 1턴
선행 조건: 신호 진단
```

### 미확인 구조 신호 확인

```text
전력 1
소요 1턴
성공 시 보상 큼
실패 시 항로 지연
```

### 방사선 차폐 강화

```text
수리 부품 2
선행 조건: 외벽 손상 조사
다음 사건 피해 감소
```

## 6.7 임무 유형

- 일반 임무
- 마감 임무
- 선행 조건 임무
- 여러 턴이 필요한 지속 임무
- 같은 태그를 연속 처리하면 이득을 받는 임무
- 위험·보상이 큰 선택형 임무
- 긴급 임무
- 필수 임무

## 6.8 라운드 흐름

### 1단계: 상황 공개

현재 자원과 임무 목록을 보여준다.

### 2단계: 계획

플레이어가 수행 순서를 지정한다.

### 3단계: 실행

행동을 하나씩 실행한다.

### 4단계: 사건

특정 시점에 사건 카드가 발생한다.

### 5단계: 대응

기존 계획을 유지하거나 변경한다.

### 6단계: 정산

완료 임무, 실패 임무, 남은 자원, 안정도, 점수를 계산한다.

## 6.9 사건 카드 유형

### 긍정 사건

- 비상 전력 셀 발견
- 행동 포인트 회복
- 임무 마감 연장
- 특정 수리 비용 감소

### 부정 사건

- 태양풍
- 기관실 과열
- 승무원 부상
- 전력 계통 고장
- 긴급 임무 추가
- 특정 행동 일시 차단

### 선택형 사건

```text
구조 신호가 수신되었습니다.

A. 구조를 시도한다.
전력 -2 / 성공 시 항해 평판 상승

B. 기존 임무를 유지한다.
선체 안정도 유지 / 구조 임무 포기
```

## 6.10 라운드별 난이도

| 라운드 | 상황 | 추가 요소 |
|---|---|---|
| 1 | 출항 점검 | 기본 임무와 자원 사용 |
| 2 | 통신 장애 | 중요도 차이 |
| 3 | 소행성 지대 | 마감 도입 |
| 4 | 기관실 과열 | 선행 임무 |
| 5 | 우주 폭풍 | 사건 카드 |
| 6 | 승무원 부상 | 승무원 제약 |
| 7 | 외계 신호 | 위험·보상 선택 |
| 8 | 항법 시스템 붕괴 | 모든 요소 종합 |

## 6.11 통과 기준

- 필수 임무 성공
- 선체 안정도 1 이상
- 라운드별 최소 점수 이상

예시 최소 점수:

| 라운드 | 최소 점수 |
|---|---:|
| 1 | 500 |
| 2 | 800 |
| 3 | 1,100 |
| 4 | 1,400 |
| 5 | 1,700 |
| 6 | 2,000 |
| 7 | 2,300 |
| 8 | 2,600 |

실제 값은 플레이테스트 후 조정한다.

## 6.12 COMMAND DECK 성공률

단순 임무 개수 대신 중요도 가중치를 사용한다.

```text
높음 = 3
보통 = 2
낮음 = 1
```

```text
성공률 =
완료한 임무 중요도 합
÷ 전체 임무 중요도 합
× 100
```

# 7. GAME 2: CARGO CORE

## 7.1 한글명

화물 분류실

## 7.2 게임 목적

행성 표본, 광물, 연료 셀, 생물 화물을 규칙에 따라 올바른 보관 구역으로 분류하는 작업기억 게임.

## 7.3 주요 훈련 요소

- 작업기억
- 규칙 유지
- 예외 처리
- 방해 정보 억제
- 규칙 우선순위 적용
- 정보 갱신

## 7.4 최대 라운드

**10라운드**

예상 플레이 시간:

```text
전체 진행 12~18분
```

## 7.5 화물 속성

- 출신 행성
- 색상 코드
- 화물 종류
- 위험 등급
- 중량
- 온도
- 생체 반응 여부
- 특수 아이콘

## 7.6 분류 구역

- 일반 화물칸
- 냉동 보관실
- 위험물 격리실
- 연구실
- 동력실
- 우주 폐기 구역
- 임시 보류 구역

## 7.7 규칙 예시

```text
청색 연료 셀은 동력실로 보낸다.

생체 반응이 있는 표본은 연구실로 보낸다.

위험 등급 3 이상은 다른 규칙보다 먼저 적용하여 격리실로 보낸다.

단, 동결 표시가 있는 생체 표본은 냉동 보관실로 보낸다.
```

## 7.8 규칙 유형

- 단일 규칙
- 두 속성 조합 규칙
- 예외 규칙
- 우선순위 규칙
- 반전 규칙
- 직전 화물 기억 규칙
- 연속 조건 규칙
- 라운드 중 일부 규칙 교체

## 7.9 규칙 우선순위

충돌할 경우 반드시 다음 순서로 처리한다.

```text
1. 특수 규칙
2. 예외 규칙
3. 조합 규칙
4. 일반 규칙
```

예:

```text
일반: 청색 화물은 일반 화물칸
예외: 위험 등급 3 이상은 격리실

청색 + 위험 등급 3
→ 격리실
```

## 7.10 라운드 흐름

```text
규칙 공개
→ 규칙 확인 시간
→ 규칙창 접힘
→ 화물 연속 등장
→ 분류 입력
→ 라운드 결과
```

## 7.11 라운드별 난이도

| 라운드 | 규칙 수 | 화물 수 | 주요 변화 |
|---|---:|---:|---|
| 1 | 1 | 8 | 색상 기준, 규칙 항상 표시 |
| 2 | 2 | 10 | 화물 종류 추가 |
| 3 | 2 | 10 | 규칙창 접힘 |

# 9. 진척도 시스템

## 9.1 전체 라운드 수

## 9.2 게임별 진척도

```text
게임별 진척도 =
해당 게임에서 통과한 최고 라운드
÷ 해당 게임 전체 라운드
× 100
```

예:

```text
COMMAND DECK
6 / 8 통과

```

## 9.3 전체 진척도

```text
전체 진척도 =

÷ 30
× 100
```

- 라운드에 진입한 것만으로 증가하지 않는다.

- 재도전에서 낮은 점수가 나와도 기존 최고 진척도는 내려가지 않는다.
- 다른 기기 기록과 충돌하면 높은 진척도를 유지한다.

# 12. 사용자 계정 정책

## 12.1 로그인 방식

```text
닉네임 + 4자리 숫자 PIN
```

## 12.2 확정 정책

- 닉네임 중복 불가
- 대소문자 구분 없이 중복 처리
- 닉네임 변경 불가

- PIN 재설정 없음
- 이메일 입력 없음
- 같은 브라우저에서는 자동 로그인
- 다른 기기에서는 닉네임과 PIN으로 로그인
- PIN을 잊으면 기록 복구 불가

## 12.3 닉네임 규칙

- 2~12자
- 한글, 영문, 숫자 허용
- `_`, `-` 허용
- 앞뒤 공백 제거
- 연속 공백 금지
- 대소문자 구분 없이 중복 처리
- 욕설 및 금칙어 필터 적용

예:

```text
NOVA
Nova
nova
```

세 닉네임은 동일한 것으로 처리한다.

## 12.4 등록 화면 고지

```text
닉네임은 등록 후 변경할 수 없습니다.
PIN을 잊으면 진행 기록을 복구할 수 없습니다.
```

## 12.5 게스트 플레이

등록하지 않고 게임을 시작할 수 있다.

게스트 기록은 현재 브라우저에만 저장한다.

나중에 회원 등록 시 게스트 기록을 계정에 병합한다.

# 13. 인증 및 세션

## 13.1 PIN 저장

PIN 원문을 데이터베이스에 저장하지 않는다.

권장 방식:

```text
PBKDF2
사용자별 랜덤 salt
SHA-256
충분한 반복 횟수
```

4자리 PIN은 가능한 값이 10,000개뿐이므로 강한 인증 수단은 아니다. 이 서비스에는 게임 기록 외 민감한 개인정보를 저장하지 않는다.

## 13.2 자동 로그인

등록 또는 로그인 성공 시 무작위 세션 토큰을 생성한다.

```text
브라우저:
세션 토큰 원문을 HttpOnly 쿠키에 저장

D1:
세션 토큰 해시만 저장
```

권장 쿠키:

```text
HttpOnly
Secure
SameSite=Lax
Path=/
Max-Age=2592000
```

기본 유효기간은 30일.

## 13.3 로그인 실패 제한

권장 정책:

```text
10분 안에 5회 실패
→ 일정 시간 로그인 차단
```

MVP에서는 D1에 실패 횟수와 잠금 시각을 저장하는 방식으로 구현한다.

# 14. 기록 자동 저장

## 14.1 로컬 저장

게임 중에는 각 행동 후 브라우저에 현재 상태를 저장한다.

권장:

- IndexedDB
- 간단한 경우 localStorage

저장 항목:

- 현재 게임
- 현재 라운드
- 현재 문제 인덱스
- 현재 점수
- 성공 수
- 전체 시도 수
- 남은 시간
- 미전송 결과 목록

## 14.2 서버 저장

서버에는 매 행동마다 요청하지 않는다.

다음 시점에만 저장한다.

- 라운드 통과
- 라운드 실패
- 게임 종료
- 재접속 후 미전송 결과 동기화

## 14.3 오프라인 대응

```text
라운드 종료
→ 로컬에 결과 저장
→ 서버 전송
→ 실패 시 pendingSync에 유지
→ 다음 접속 또는 다음 라운드 종료 시 재시도
```

사용자 표시 문구:

```text
항해 기록이 기기에 저장되었습니다.
연결이 복구되면 서버와 자동 동기화됩니다.
```

## 14.4 중복 저장 방지

각 라운드 결과에 고유 `sessionId`를 부여한다.

같은 결과가 여러 번 전송되어도 한 번만 반영한다.

# 20. 서버 검증

클라이언트가 최종 진척도나 성공률을 직접 제출하지 않는다.

서버가 원시 결과를 받아 계산한다.

검증 항목:

- 유효한 gameId
- 유효한 roundNumber
- 현재 잠금 해제된 라운드인지
- attempts와 해당 라운드 문제 수가 일치하는지
- successes가 attempts 이하인지
- 가능한 최대 점수를 넘지 않았는지
- 플레이 시간이 비정상적으로 짧지 않은지
- sessionId가 중복인지
- 이전 라운드 통과 여부

예:

```text
CARGO CORE 5라운드
문제 수 12

attempts != 12 → 거부
successes > 12 → 거부
durationMs < 최소 시간 → 거부
4라운드 미통과 → 거부
```

완전한 치팅 방지보다 비정상 기록 차단을 목표로 한다.

# 21. 랭킹 계산 방식

서버가 `game_progress`와 최근 `play_sessions`를 기준으로 계산한다.

```text
전체 진척도 =
(min(commandHighest, 8)
+ min(cargoHighest, 10)
+ min(vectorHighest, 12))
÷ 30 × 100
```

```text
최근 성공률 =
최근 10개 라운드 successes 합
÷ attempts 합
× 100
```

```text
랭킹 점수 =
전체 진척도 × 0.6
+ 최근 성공률 × 0.4
```

클라이언트가 랭킹 점수를 저장하지 않는다.

# 22. 데이터 보존 정책

D1 용량 절약을 위해 상세 플레이 기록을 무한히 보존하지 않는다.

권장:

- 사용자당 최근 플레이 30개 유지
- 또는 30일이 지난 상세 기록 삭제
- 누적 성공 수와 시도 수는 `game_progress`에 유지
- 최고 라운드와 최고 점수는 영구 유지

MVP에서는 자동 정리 작업 없이 수동 SQL 정리도 가능하다.

# 29. 구현 시 가장 중요한 원칙

1. 게임은 LLM이나 외부 API 없이 완결되어야 한다.
2. 게임 규칙과 정답은 프론트와 서버에서 각각 검증 가능해야 한다.
3. 서버에는 매 행동이 아니라 라운드 종료 결과만 저장한다.
4. 사용자의 실패를 질환이나 능력 부족으로 해석하지 않는다.
5. 닉네임, 진척도, 성공률, 랭킹 외 불필요한 개인정보는 수집하지 않는다.
6. 의료행위나 치료 효과를 암시하지 않는다.
7. 서버 저장 실패가 게임 플레이 실패가 되지 않도록 한다.
8. 세 게임의 세계관과 UI는 하나의 우주선 안에서 일관되게 유지한다.
9. 모바일 조작성을 데스크톱과 동등하게 고려한다.
10. 랭킹보다 게임 자체의 재미와 재도전성을 우선한다.
